# STYLE GUIDE — vinci_world

Status: AVAILABLE（v1.0，2026-07-04 — USER 定案：中間檔要豐富，不要縮寫）

本檔約束**所有**寫進 `<STORY_ROOT>` 的文字——設計文件、交接檔、審查檔、
劇本、對白，一體適用。它是 WORKFLOW_CORE_VARIABLES「不發明中文詞彙」的
執行細則：適用範圍明確涵蓋**設計行文**，不只世界內用語。

## 1. 交接檔＝富文本，不是摘要

- 每一份中間檔都寫給「下一位完全沒讀過前文的同事」看：完整的句子、
  講清楚來龍去脈。下游 worker 靠的是去檔案裡大量查閱，不是靠標籤解壓縮。
- 每條要下傳的約束，寫齊三件事：**規則本身**（白話說清楚）、**出處**
  （哪份檔案、哪一次定案）、**違反時長什麼樣**（給一個具體例子）。
  一條約束就是一小段話——寧可超額，不可縮寫。
- 意思可以重複出現，但用詞要多樣；不要把同一句速記當咒語逐字複誦。
  豐富而多樣的內容，勝過短而神祕的代號。

## 2. 禁止造詞（設計行文與故事行文一體適用）

- **不得新造**壓縮標籤、黑話、代號來指稱約束、節拍或紀律。
- 遇到上游檔案已經造出來的（例：五拍、彎點、回聲拍、門檻拍、一滴、
  防走樣、加深不加倍、帳直、口傳樞紐、O1–O4、L1–L6 這一類），下傳時
  **展開成白話**並註明出處；原標籤只能以引用形式出現在表格欄位或
  出處註記裡，不得進入行文與敘述。
- **Canon 白名單**（世界基線已通過驗收、屬於定案設計詞，可直接使用）：
  常識層／線索層／無人層；露天層／室內層／門縫層；彼端；
  投影／投影法則／收藏家／原件／點數；渡口／渡進／渡回。
- 對白另受既有紀律：不得出現現實技術詞（區塊鏈、上鏈、NFT、token、
  錢包⋯⋯）；台式口語自然；角色工作名未經 USER 定案前不得在對白中
  當成已確認的名字使用。

## 3. 劇本形式（USER 定案 2026-07-04）

vinci_world 的章節劇本（chapter STEP 6 的產物）用**通用劇本形式**。第二人稱
「你推門、你跑起來」那種文字遊戲體是 rpg-1 的落地形式，本專案不用——這裡的
落地面是過場動畫，劇本要讓讀的人看見畫面：

- **場景標頭**：場號｜時段｜地點。
- **舞台指示**：第三人稱、現在式，只寫畫面上看得到、聽得到的事——誰在哪、
  做什麼動作、鏡頭與聲音的提示。玩家角色一律稱「玩家」。
- **對白行**：說話者（職稱）＋台詞。玩家的內心話寫成「玩家（內心）：⋯」，
  對應 runtime 掛名的獨白。
- 拍不出來的感官敘述（聞到什麼、摸起來如何、心裡暖暖的）不寫進舞台指示；
  要傳達就轉成畫面、聲音，或放進內心台詞。

## 3.1 劇本與敘述

- 敘述句不得帶設計公式腔。判斷方法：一句話如果讀起來像在替系統下註解
  （例：「聽得見、講得開，帳只有一個去處」），就刪掉或改寫成場面上的
  人話——用動作、位置、聲音、物件說同一件事。
- 台式書面、對白口語自然、避免翻譯腔（承 WORKFLOW_CORE_VARIABLES 的
  創作語言條款）。

## 4. 禁 LLM 腔（USER 定案 2026-07-04）

寫出來的中文要像台灣人會講、會寫的話。以下是常見病徵，reviewer 拿著這張
清單對，看到就要求改寫：

- **自創名詞化稱謂**：「進入者」「守望者」這類「X＋者」的造詞（既有的
  正常詞如「見證人」「收藏家」不在此限）。
- **翻譯腔句式**：「在他眼前展開」「作為一個⋯」「是⋯的存在」「進行⋯」
  這類從英文直譯過來的骨架。
- **破折號連鎖**：用「——⋯——⋯」當句子的主幹。破折號一句最多一組。
- **比喻堆疊**：一句話裡疊兩個以上比喻（例：「主旋律＋底色＋心跳」擠在
  同一句）。一句一個比喻，用完就落地。
- **土法檢測**：把句子讀出聲。讀起來不像人講話，就改。

## 5. Reviewer 的驗法

- 驗的是**意思有沒有走樣**：回到上游原文比對語義，不是數標籤數量、
  不是 grep 代號有沒有出現。
- 在任何新產物裡看到新造的壓縮詞，直接 FAIL，理由寫明是哪個詞、
  應該展開成什麼樣的白話。
