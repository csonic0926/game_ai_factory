# Choice Aftermath Writing

*為「某個選項的後續」設計並撰寫分支劇情（以節奏為主，通常 10–18 行；必要時可到 8–24），以 `<PRIMARY_LOCALE>` 為主、所有 `<SHIPPED_LOCALES>` 皆為自然文案，最後依 adapter 的 `LANDING_SPEC.md` 落地。*

用戶只要提出「某個選項的後續」（例如：進入客棧後，立即上樓 / 避事）就必用此 skill。

本 skill 的目標是：
先把「分支劇情」設計清楚（可回到主線、差異足夠大、至少一個後果），
再依 adapter 的 `LANDING_SPEC.md` 把內容落到 runtime 資料，做到可直接進遊戲測試。

---

## 寫作規格（硬規則）

* 語言：以 `<PRIMARY_LOCALE>` 為主；同時產出所有 `<SHIPPED_LOCALES>`，每個語言都要**自然文案**（非機械直譯）。（example for a zh-TW/en/ja project：繁中（台式）為主，同時產出 en/ja。）
* 敘事視角：第二人稱（以「你」稱呼玩家角色／主角）。
* 對白格式：沿用專案既有台詞格式（多為直接引號台詞，例如 `「……」` / `"..."`），
  使用動作或語氣補強對白承接，維持格式一致。
* 每個分支至少要有 1 個「後果」：情報 / 資源 / 關係 / 伏筆 / 時間風險，擇一即可。
* 分支結尾提供 choice，且選項差異足夠大（行動、風險或立場上的差異）。

---

## 篇幅與停止條件（v1.2 修正版，硬規則）

* **篇幅判斷唯一依據**：
  敘事推進到「主角可以做出合理、且有差異的多個行動選項」。
* **行數上限**：硬上限 **24 行**。
* **行數下限**：不設硬下限；但正文需清楚交代：

  * 主角所在位置與狀態
  * 當前局勢與風險
  * 為何此刻需要玩家做選擇
* **一旦出現合理的多個行動選項，即停止正文並進入 choice**；
  保持節奏，不為了湊行數延寫。

### 單一分支內的節奏原則

* 每行只承載一個鏡頭 / 動作 / 感受。
* 重大事件（踹門、冷笑、亮兵器、出手）前後，各保留至少 1 行作為呼吸。
* 本分支內容以自身節奏為準，保持清楚的鏡頭推進。
* `<PRIMARY_LOCALE>` 文案若已達現在的自然度，但單行在遊戲內讀感偏滿，優先拆成 2 行短節拍；不要只因為句子通順就把兩個節拍擠在同一 click。

### 同層分支的正確用途（嚴格限定）

* 參考同層已完成分支，僅用於：

  * 同步時間點（共通事件進展到哪一步）
  * 同步世界狀態（誰已出現、誰被點名、局勢是否升溫）
  * 避免敘事互相矛盾
* 以同層分支作為一致性參照，保持狀態同步即可。

### 【v1.3 新增】同層分支的「結構分析」（zero-shot，必做）

> 目的：避免新增分支在劇情邏輯上沒有空間、或只是換場景但不換結果。
> 本段要求你**不用依賴用戶提供分析**，只靠既有分支與最小上下文自行推導（zero-shot）。

* 先用一句話為每個同層分支標註「功能」：例如 `回避` / `介入` / `衝突` / `逃脫` / `交易` / `揭露` / `背刺`。
* 再用一句話寫出它的「不可替代性」：為什麼只有這個分支能帶出這個後果/節奏。
* 若你發現新分支（例如「去後廚」）在既有局勢下**沒有合劇情的空間**：

  1) 直接承認「空間不足」，不要硬補常規結果。
  2) 改採「跳脫解法」：引入第三方/隱藏機關/誤會與錯認/反轉責任，讓分支改變的是**局勢框架**而非戰力。
  3) 跳脫解法仍必須滿足：可回主線、至少一個後果清楚、與同層分支結果不重疊。

* 產出時，先給 1 句「分支定位」說明，再開始寫正文（避免寫到一半才發現分支站不住）。

### 【v1.4 新增】暗線保密層級與掩護人設（必做）

> 目的：避免把「暗線」寫成「明線」，或讓暗部 NPC 在不該的情境主動爆料。
> 核心：**資訊是否能說，取決於主角此刻的路徑與身份**，而不是取決於作者（或玩家）之後會知道什麼。

* 在動筆前，先宣告本分支的 `SECRECY_LEVEL`（預設 `COVERT`）：

  * `COVERT`：不說組織/家族/名單/真名；只用「規矩」「生意話」「避事話」與行為去讓玩家感覺不對勁。
  * `PARTIAL`：最多允許 1 個「外人也可能聽過的」代稱或碎片（例如「青衣」「墨味」這類描述），仍不給可追溯的關鍵名詞。
  * `OPEN`：只有在同一 TIMEBOX 的既有分支已經公開、且主角此刻合理可知時，才允許真名/勢力名。

* **掩護人設優先**：暗部 NPC 面對「未驗明」的主角時，必須先維持表面人設。

  * 先做：趕人/立規矩/只談商品/只談風雪與病痛。
  * 不做：主動交代內情、點名誰在客棧、講「名單/點人/封家」等會把線索變成明牌的詞。

* **路徑差異不等於知情共享**：

  * 若同層另一路徑（例如走地道）讓主角看到更深一層，該資訊**不得**自動回填到「從街口闖入」的版本。
  * 同一 NPC 可以在不同路徑呈現不同深度，但必須能用「驗明/暗號/熟門熟路」去解釋。

* **暗示要可回收，但不可定錨**：

  * 可以讓玩家事後回想「原來那句話是這個意思」。
  * 但不要在當下丟出可被玩家立刻命名與整理的明確結論（例如真名、家族、名單）。

* 寫完自檢（必做）：

  * 把本段所有台詞與敘事掃一遍，確認沒有「真名/勢力名/名單/點名」等定錨詞。
  * 將可能爆線的句子改成：行為控場 + 生意話 + 規矩話（讓玩家感覺被分類處理）。

### 【新增】同層分支的 Battle ID 一致性（必做）

當分支會觸發戰鬥時：

- 先定位該分支實際觸發的戰鬥 id（事件 → 戰鬥資料的對應方式，依 adapter 的 `LANDING_SPEC.md`；戰鬥資料本身見 `<BATTLE_SYSTEM>`，若為 `NOT_AVAILABLE` 則本節整段跳過）。
- 讀該戰鬥的資料，確認這場戰鬥的「事實」：

  - 是單人還是多敵（有沒有佈陣/layout）
  - 參戰盟友是否存在（分支是否會 recruit / 隊伍編成是否固定）

- 你寫的敘事必須對齊同一個戰鬥 id 的事實（不能同 id 在不同分支變成另一場戰鬥）。
- 若你發現同一戰鬥 id 被多個分支共用，但各分支需要不同戰鬥事實：

  1) 優先建新戰鬥 id（例如 `_SOLO` / `_OPTION_2` / `_WITH_FENG`）。
  2) 然後回來讓分支敘事對齊各自的戰鬥 id。


## 【v1.1 新增】中文母語穩定度規則（非常重要）（example for a zh-TW primary locale — Chinese language craft）

> 本 skill **優先確保「像人寫的中文」**，再談節奏。
> **可以省資訊，但不能省語法骨架。**

### A) 功能詞骨架不可硬刪

在調整句子與去寫作腔時，**保留維持中文自然性的功能詞骨架**，包含但不限於：

* 行為承載詞：
  `說 / 開口 / 笑 / 看 / 停住 / 抬頭 / 迎上去`
* 語氣與承接詞：
  `而 / 卻 / 仍 / 還 / 卻又`
* 行為完成標記（必要時）：
  `了 / 著 / 過`

📌 判斷準則：

> 若刪除後句子「語意仍懂，但不像母語者會這樣寫」，
> **寧可稍長，也要補回功能詞。**

---

### B) 人物發聲必須「顯性發聲」

凡出現人物話語，必須符合其一：

* 有明確的發聲動作
* 或有清楚的引語結構

✅ 讓發聲行為或引語結構清楚呈現，並在敘事與台詞之間建立可讀的承接。

---

### C) 狀態描寫優先使用「既有漢語搭配」

使用既有漢語搭配描寫狀態，保持自然語感與可讀性。

## 驗收清單（完成後必查）

* `<PRIMARY_LOCALE>` 讀起來像母語者敘事，無「怪省略」或外文化句法
* `<PRIMARY_LOCALE>` 單行若偏密，已優先拆成 2 行短節拍，而不是維持 1 行長句
* 所有 `<SHIPPED_LOCALES>` 同 key、同節奏對齊
* 至少 1 個後果清楚可見
* 分支選項差異夠大
* 無重複 key / id（全 repo 搜索確認）

## Reference implementation: event-timeline CSV runtime (rpg-1 heritage)

> 以下是本文件原始 rpg-1 版本的具體落地細節，僅作為 worked example 保留。
> 真實專案一律依各自 adapter 的 `LANDING_SPEC.md` 落地。

- 落地目標：`settings/event*.csv` + `locales/locales.csv`；預設繁中（台式）+ en/ja。
- 落地執行：交給 `story-event-csv` skill（rpg-1 專用）。
- 承接點定位輸入的 id 範例：
  - 選項文字 key：`story_telling.prologue_inn_choice_upstairs`
  - `event_id`：`EV_ROUTE_*` / `EV_STORY_*`
  - `world_map_node` id：`flow_*`
- Battle ID 一致性的具體對應：
  - 分支觸發戰鬥＝`settings/event.csv` 的 `type=BATTLE`。
  - 從 `event_id → settings/event.csv → battle_id` 定位。
  - 讀 `settings/battles.csv` +（若有）`settings/battle_layouts.csv`，確認單人/多敵（有沒有 `battle_layout_id`）與盟友（分支是否 recruit / `party_loadouts` 是否固定）。
- Story 事件現在會自動 append；不要再依賴 `append_story_profile` 欄位。
